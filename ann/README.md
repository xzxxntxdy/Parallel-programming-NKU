---

## 1. 选题背景

最近邻搜索（Nearest Neighbor Search, NNS）的目标是在高维空间中找到与查询向量最接近的向量。

由于深度学习等技术可以将图像、文本、音频等非结构化数据转换为高维向量，并尽量保持语义相似性，例如相似图片对应的向量距离也较近，因此 NNS 被广泛应用于推荐系统、搜索引擎和大语言模型等场景。

在实际应用中，向量规模通常大于千万级，向量维度通常也大于 100 维。普通精确搜索算法复杂度过高，难以直接应用。因此，实际向量检索系统通常使用近似最近邻搜索（Approximate Nearest Neighbor Search, ANNS）。ANNS 的目标是在保持较高准确率的同时降低查询时间。

---

## 2. 问题定义

给定一个 `d` 维向量数据集合：

```math
X = \{x_1, x_2, x_3, \dots, x_N\} \subset \mathbb{R}^d
```

用 `δ(x, y)` 表示向量 `x` 和向量 `y` 的距离。给定查询向量：

```math
q \in \mathbb{R}^d
```

`kNNS` 的目标是找到集合 `S`，满足：

```math
|S| = k, \quad S \subseteq X
```

并且：

```math
\forall x \in S, \forall y \notin S, \quad \delta(x, q) < \delta(y, q)
```

直观理解：`S` 是 `X` 中距离查询向量 `q` 最近的前 `k` 个向量组成的集合。

ANNS 不要求返回集合中的 `k` 个向量都是真实最近的 `k` 个向量，只要求达到一定准确度即可。

ANNS 算法通常用以下两个指标衡量：

- 查询延时（latency）
- `recall@k`

`recall@k` 定义为：

```math
recall@k = \frac{KNN(q) \cap KANN(q)}{k}
```

其中：

- `KNN(q)` 表示查询的精确答案集合。
- `KANN(q)` 表示使用 ANNS 找到的答案集合。

比较 ANNS 加速比时，通常比较在达到某一精度时不同算法的延时。例如：比较达到 `recall@100 = 0.95` 时的 latency。

## 3. 算法方向与实验重点

ANNS 算法的优化可以分为两个方面：

1. 优化距离计算速度，即加快 `δ(x, y)` 的计算。
2. 减少需要访问的数据点数量。普通算法对每个查询都遍历 `X` 中的每个点，而好的 ANNS 算法应尽可能减少访问点数量。

ANNS 算法通常分为两个阶段：

- 索引构建：离线阶段。
- 查询：在线阶段。

本学期并行实验重点关注**查询性能优化**。如果对索引构建性能有好的优化算法，也可以写在报告中。

## 3.1 SIMD

SIMD 实验重点关注**距离计算速度优化**。

常见距离包括欧几里得距离和内积距离。

欧几里得距离定义为：

```math
\delta(x, y) = \sum_{i=1}^{d}(x_i - y_i)^2
```

内积距离定义为：

```math
\delta(x, y) = 1 - \sum_{i=1}^{d} x_i \cdot y_i
```

朴素 SIMD 优化是使用 SIMD 指令加速上述计算公式。

进一步的距离计算优化方法是 Vector Quantization（VQ）。VQ 又分为：

- Scalar Quantization（SQ）
- Product Quantization（PQ）

### 3.1.1 SQ

原始向量通常每一维都用一个 32-bit 浮点数存储。假设使用 128-bit SIMD 寄存器加速计算，最多一次计算 4 个浮点数。

SQ 可以将向量每一维量化为 8-bit 无符号整数，此时使用 128-bit 寄存器计算时，可以一次计算 16 个 `uint8`。

PDF 中给出的 SQ 简易参考资料为：

```text
https://zhuanlan.zhihu.com/p/719012156
```

### 3.1.2 PQ

以 `N` 条 128 维向量为例，每条向量由 128 个浮点数组成：

```text
(x0, x1, ..., x126, x127)
```

PQ 的基本过程如下。

#### 第一步：向量切分

将每条向量切分为 4 段：

- `(x0, ..., x31)` 为第一段。
- `(x32, ..., x63)` 为第二段。
- `(x64, ..., x95)` 为第三段。
- `(x96, ..., x127)` 为第四段。

#### 第二步：子空间聚类

将四个子段视作四个独立子空间，分别对切片得到的子向量进行聚类：

- 对 `N` 个 `(x0, ..., x31)` 聚类。
- 对 `N` 个 `(x32, ..., x63)` 聚类。
- 对 `N` 个 `(x64, ..., x95)` 聚类。
- 对 `N` 个 `(x96, ..., x127)` 聚类。

每一段聚类得到 256 个类中心，总共保存：

```text
256 × 4 = 1024
```

个类中心，并按顺序编号。

#### 第三步：量化编码

对原始向量进行量化编码。将原始向量的四个段分别用距离最近的聚类中心编号表示。

例如第一条向量的第一段 `(x0, ..., x31)` 距离第一段的第 124 个类中心最近，则将这一段 32 个浮点数替换为整数 `124`。其他段以此类推。

最终，一条 128 维浮点向量会被压缩为 4 个整数。

### PQ 查询距离计算方式

当一个查询向量到达时，计算其与 PQ 压缩后的 Base 向量之间的距离主要有两种策略：

#### 1. 对称距离计算 SDC

SDC（Symmetric Distance Computation）会对查询向量执行同样的量化编码。

过程：

1. 将查询向量切分为 4 个子向量。
2. 分别计算每个子向量与对应 256 个类中心的距离。
3. 用最近中心的 ID 编码查询向量，例如得到 `(25, 82, 10, 254)`。
4. 查询向量与 Base 向量之间的距离转化为查阅预先计算好的 `256 × 256` 中心间距离表。
5. 对所有 Base 编码进行 `4 × N` 次查表累加。

#### 2. 非对称距离计算 ADC

ADC（Asymmetric Distance Computation）不对查询向量进行编码。

过程：

1. 查询到达时，直接计算 4 个原始查询子向量到对应子空间 256 个类中心的精确距离。
2. 生成一张 `4 × 256` 的局部查找表 LUT。
3. 对 Base 编码在该 LUT 中进行 `4 × N` 次查表累加，得到查询距离。

相比 SDC，ADC 的优势是：

- 仅引入 Base 端的量化误差。
- 召回率更高。
- 无需维护庞大的中心间距离表。
- 在实时查询，即单条触发场景下，两者总计算量相当，但 ADC 往往具有更好的性能权衡。

### SDC 的流水线并行探索

在批量查询场景下，SDC 方案具有流水线优化潜力。

由于 SDC 涉及查询向量的量化编码，可以尝试“生产者-消费者”模型：

- 使用独立线程异步编码下一批查询向量。
- 搜索线程并行执行当前批次的查表累加。
- 也可以尝试分配独立线程进行结果重排。

通过流水线设计，可以掩盖查询编码阶段的延迟。这种结合多线程实现不同阶段重叠运行的方式，是后续实验中值得尝试的进阶方向。

## 3.2 pthread / OpenMP

后续实验重点关注如何减少需要访问的数据点数量。

一种朴素方法是将 `X` 中的向量划分到各个线程中，使每个线程只访问一部分向量。

### 3.2.1 IVF 和 IVF-PQ

IVF（Inverted File）在预处理阶段对 `X` 中的向量进行 KMeans 聚类。

对于每个查询：

1. 检索距离最近的 `m` 个聚类簇。
2. 这一步中，用聚类中心到查询的距离代替簇中其他点到查询的距离。
3. 遍历这 `m` 个聚类簇下的所有候选。
4. 计算得到距离最近的 top-k 候选。
5. 遍历 `m` 个簇时，可以使用并行加速。

通常可以通过调整 `m` 提高 `recall@k`，但同时会提高 latency。这体现了 latency 与 `recall@k` 之间的 tradeoff。

实验报告中需要用图表体现：

- 不同算法的 latency-recall@k tradeoff。
- 不同线程数的 latency-recall@k tradeoff。

IVF-PQ 是将 IVF 和 PQ 结合起来。PDF 中给出的参考资料为：

```text
https://zhuanlan.zhihu.com/p/378725270
```

### 3.2.2 更多探索：图索引

本节技术难度较大，学有余力的同学可以自主探索。

目前真正主流的 ANNS 算法是基于图的方法，其在各种场景下性能远高于其他算法。目前工业界应用最广的图索引为 HNSW。原论文为：

```text
Efficient and robust approximate nearest neighbor search using Hierarchical Navigable Small World graphs
```

探索图索引算法时，可以直接使用 HNSW 开源库 `hnswlib`，并在其基础上开发。

图索引的并行优化难度较大。由于 HNSW 搜索路径有强数据依赖性，实现高效多线程并行具有较大挑战。可以从以下三个方向尝试和讨论：

#### 方向 1：边划分并行

在搜索过程每一步，当需要计算当前节点所有邻居到查询向量的距离时，将这些邻居，即边，分配给多个线程并行计算。

特点：

- 逻辑实现简单。
- 当邻居数量 `M` 较小时，线程创建与同步开销可能抵消计算收益。

#### 方向 2：Layer 0 点划分并行

针对 Layer 0 搜索时维护的 `efSearch` 大小候选集合实现多线程协同搜索。

做法：

- 多个线程同时从全局候选队列中取出不同节点。
- 并行探索各自邻居。
- 更新全局结果集。

该方法需要设计高效线程同步机制，以处理并发读写。

#### 方向 3：多入口点并行

启动多个独立线程，从不同入口点或随机起始点同时发起多次完整搜索。

每个线程独立执行搜索流程，最终在主线程中对各线程搜到的 Top-K 结果进行合并排序。

该方法虽然不直接减少单次搜索耗时，但能通过增加搜索覆盖面显著提升召回率。

## 3.3 MPI

MPI 的整体优化思路与多线程大致相似。

可以自由探索更多方法，例如：

- 使用 MPI 实现 IVF + PQ。
- 使用 MPI 实现 IVF + HNSW。
- 使用 MPI 实现 HNSW + HNSW。

其中 HNSW + HNSW 指两层 HNSW：

- 第一层 HNSW 中的每个点代表一个小 HNSW。
- 查询时进行嵌套搜索。

## 3.4 GPU

GPU 优化通常以提高查询吞吐率为主，即 Query 间并行。

可考虑的划分方式包括：

- 对查询集进行划分。
- 对数据集进行划分。
- 每个线程块负责一部分查询任务或数据集任务。

也可以结合 GPU 架构特点，设计其他 GPU 并行算法。

此外，ANNS 的计算也可以转换为矩阵乘法计算。同学们可以自行进行相关公式推导，并编写 GPU 加速代码来加速矩阵乘法。
