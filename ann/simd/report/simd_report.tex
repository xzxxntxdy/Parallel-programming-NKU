\begin{abstract}
\small
本文针对DEEP100K数据集（$d=96$, $N=100000$, 内积距离），系统实现并深入分析了近似最近邻搜索（ANNS）
查询阶段的六类SIMD加速方案：Flat-SIMD、标量量化（SQ-SIMD）、乘积量化配合对称/非对称距离计算
（PQ-SDC/PQ-ADC）、SDC流水线并行以及FastScan查表优化。
实验覆盖ARM Kunpeng-920（NEON 128-bit）和Intel Core i9-14900HX（AVX2 256-bit）双平台的全部强制实验矩阵，
所有数据经真实运行验证。
最终方案FastScan v2（lane-major布局+寄存器累加+4路展开）在ARM上以2.44~ms/query
（recall@10=0.99995, 7.82$\times$加速），
在x86上以1.06~ms/query（recall@10=1.0, 3.36$\times$加速）取得全局最优。
本文详细记录了从Flat-Scalar到FastScan v2的完整优化路径，重点分析了FastScan从v1到v2的
代码布局优化过程及其背后的cache/microarchitecture原理，通过perf计数器与汇编分析验证优化效果。
\end{abstract}

\section{问题定义与评价指标}
% ===================================================================

\subsection{kNN与ANNS问题定义}

给定数据集$X = \{x_1, x_2, \ldots, x_N\} \subset \mathbb{R}^d$和查询向量$q \in \mathbb{R}^d$，
精确$k$最近邻（kNN）搜索的目标是找到距离$q$最近的$k$个向量。近似最近邻搜索（ANNS）不要求返回
完全精确的$k$个最近邻，而是允许以高准确率换取更低的查询延迟。本项目的目标是在保持
recall@$k\approx 1$的前提下，最大化查询吞吐或最小化单query延迟。

\subsection{距离函数与评价指标}

本项目采用框架一致的内积距离：
\begin{equation}
\delta(q, x) = 1 - \sum_{i=1}^{d} q_i x_i \label{eq:ip}
\end{equation}
内积$\langle q, x\rangle$越大，向量越相似，距离越接近0。选择内积而非欧几里得距离是因为
DEEP100K数据集已在预处理中进行了归一化，内积等价于余弦相似度。

\textbf{recall@k}定义为近似搜索结果与精确搜索结果的交集比例：
\begin{equation}
\text{recall@k} = \frac{|\text{KNN}(q) \cap \text{KANN}(q)|}{k}
\end{equation}
其中$\text{KNN}(q)$为精确搜索结果集合（由ground truth提供），$\text{KANN}(q)$为算法返回的
近似结果集合。本项目固定$k=10$。精确Flat Search的recall恒为1.0（浮点舍入误差可忽略）；
近似算法的目标是在recall$\ge 0.999$（对于最终提交）的前提下最小化延迟。

\textbf{latency}指标包括：
\begin{itemize}[nosep]
    \item 单query总延迟（从查询发起到返回top-$k$结果的墙钟时间）
    \item 对PQ类算法进一步拆分为：query编码时间、LUT构建时间（ADC）或中心间距离表时间（SDC）、
          scan查表累加时间、top-$k$/top-$p$选择时间、float精排（rerank）时间
\end{itemize}

\subsection{数据集与代码约束}

数据集为DEEP100K（图片向量），$N=100000$, $d=96$，采用内积距离。
\texttt{base}和\texttt{test\_query}均为\texttt{float*}类型，共38.4~MB。
\texttt{test\_gt}（每个query的精确top-100答案）仅用于recall评估，严禁直接作为输出答案。
框架提供朴素串行Flat Search基线：
\begin{lstlisting}
auto res = flat_search(base, test_query + i * vecdim, base_number, vecdim, k);
\end{lstlisting}
需要自行实现更优算法并替换该调用，返回值格式必须与框架一致。
新增代码文件必须使用\texttt{.h}（header-only），ARM平台通过\texttt{bash test.sh 1 1}正式运行，
不允许绕过test.sh或修改qsub.sh。

\section{SIMD算法设计}
% ===================================================================

\subsection{Flat-SIMD：朴素向量化距离计算}

\subsubsection{算法原理与数值分析}

Flat Search是最基础的ANNS查询方式：对每个query遍历全部$N=100000$个base向量，
逐对计算内积距离$1-\sum_i q_i x_i$，同时维护一个大小为$k=10$的最大堆（max-heap）
来跟踪当前$k$个最近邻。

计算量分析：
\begin{itemize}[nosep]
    \item 每query遍历$N=100000$个向量
    \item 每个向量$d=96$维，需要96次乘法和95次加法
    \item 每query总计$100000\times 96 = 9.6\times 10^6$次乘加操作
    \item 每query读取$100000\times 96\times 4 = 38.4$~MB的float base数据
    \item 算术强度 $\approx \frac{9.6\times 10^6 \text{ FLOP}}{38.4\times 10^6 \text{ bytes}} \approx 0.25$~FLOP/Byte（若按乘加各算一操作则为0.5）
\end{itemize}
极低的算术强度意味着Flat Search是典型的内存密集型（memory-bound）工作负载——
性能瓶颈不在计算单元，而在内存带宽和cache层次结构。

\subsubsection{SIMD向量化策略}

SIMD在单个指令中同时对多个数据执行相同操作，内积$\sum_i q_i x_i$天然适合向量化的批量乘加：

\begin{table}[H]
\centering
\caption{SIMD ISA维度划分（$d=96$）}
\label{tab:simd_chunk}
\begin{tabular}{lcccc}
\toprule
\textbf{SIMD ISA} & \textbf{寄存器位宽} & \textbf{float32 lanes} & \textbf{chunk/向量} & \textbf{循环迭代次数} \\
\midrule
SSE / NEON & 128-bit & 4  & 24 & $96/4 = 24$ \\
AVX / AVX2 & 256-bit & 8  & 12 & $96/8 = 12$ \\
AVX-512   & 512-bit & 16 & 6  & $96/16 = 6$ \\
\bottomrule
\end{tabular}
\end{table}

ARM NEON版本使用\texttt{fmla}（Fused Multiply-Add）指令，
单指令在128-bit Q寄存器中完成4个float的乘法和累加：
\begin{lstlisting}
float32x4_t sum = vdupq_n_f32(0.0f);
for (size_t d = 0; d + 4 <= dim; d += 4)
    sum = vmlaq_f32(sum, vld1q_f32(a + d), vld1q_f32(b + d));
float result = 1.0f - vaddvq_f32(sum);  // horizontal add on AArch64
\end{lstlisting}
NEON的\texttt{fmla}将乘加融合为单条指令，减少一半指令发射压力。

x86 AVX2版本使用4路\texttt{\_\_m256}累加器展开以隐藏乘法延迟（AGNER FOG优化法则）：
\begin{lstlisting}
__m256 s0 = _mm256_setzero_ps(), s1 = _mm256_setzero_ps();
__m256 s2 = _mm256_setzero_ps(), s3 = _mm256_setzero_ps();
for (; d + 32 <= dim; d += 32) {  // 4 chunks × 8 floats = 32
    s0 = _mm256_add_ps(s0, _mm256_mul_ps(_mm256_loadu_ps(a+d),   _mm256_loadu_ps(b+d)));
    s1 = _mm256_add_ps(s1, _mm256_mul_ps(_mm256_loadu_ps(a+d+8), _mm256_loadu_ps(b+d+8)));
    s2 = _mm256_add_ps(s2, _mm256_mul_ps(_mm256_loadu_ps(a+d+16),_mm256_loadu_ps(b+d+16)));
    s3 = _mm256_add_ps(s3, _mm256_mul_ps(_mm256_loadu_ps(a+d+24),_mm256_loadu_ps(b+d+24)));
}
// Horizontal sum: ymm → xmm → scalar
\end{lstlisting}
4路展开使x86 OoO引擎可同时飞行4条独立的256-bit乘加流水线。

理论加速比远大于实际，瓶颈在内存：每query 38.4~MB数据，ARM DDR4-2666理论带宽下限$\approx 1.8$~ms，计算再快也受限于此。

\subsubsection{编译器自动向量化 vs 手写SIMD}

编译器在\texttt{-O2}优化级别通过自动向量化（auto-vectorization）将标量循环转换为SIMD指令。
以一例展开4个累加器的自动向量化代码为例：
\begin{lstlisting}
float sum0=0,sum1=0,sum2=0,sum3=0;
for (size_t d=0; d+4<=dim; d+=4) {
    sum0 += a[d]*b[d]; sum1 += a[d+1]*b[d+1];
    sum2 += a[d+2]*b[d+2]; sum3 += a[d+3]*b[d+3];
}
\end{lstlisting}
编译器识别到循环体内4条独立乘加后，尝试用SIMD指令并行处理。
然而编译器受限于C/C++的严格别名规则（strict aliasing）和指针别名分析（alias analysis）：
如果编译器无法证明\texttt{a}和\texttt{b}不重叠，将被迫生成保守的标量代码。
手写SIMD通过\texttt{\_\_restrict}或直接使用intrinsics可以绕过这些限制。

此外，编译器自动向量化通常不会使用FMA融合（即使在支持的硬件上），
因为FMA的浮点舍入行为与分离的mul+add不同，改变舍入顺序可能违反IEEE 754标准。
手写SIMD可以显式使用\texttt{\_mm256\_fmadd\_ps}等FMA intrinsics，
前提是编译时启用\texttt{-mfma}（GCC/Clang）或\texttt{/arch:AVX512}（MSVC）。

\subsubsection{Top-k维护优化}

Flat Search需维护top-$k$最近邻。标准实现使用\texttt{std::priority\_queue}（最大堆）：
\begin{lstlisting}
if (heap.size() < k) heap.push({dist, id});
else if (dist < heap.top().first) { heap.push({dist, id}); heap.pop(); }
\end{lstlisting}
每次push/pop的复杂度为$O(\log k)$，当$k=10$时开销可接受。
此外实现了Fixed-Array TopK：维护一个大小为$k$的数组，追踪当前最差元素的下标：
\begin{lstlisting}
class FixedTopK {
    std::vector<pair<float,uint32_t>> items;
    size_t worst_idx;
    void push(float dist, uint32_t id) {
        if (items.size() < k) { items.push_back({dist,id}); recompute_worst(); }
        else if (dist < items[worst_idx].first) { items[worst_idx] = {dist,id}; recompute_worst(); }
    }
};
\end{lstlisting}
FixedTopK避免堆的$O(\log k)$调整，将push操作从分支较多的heap操作简化为$O(k)$的线性扫描。
实测中FixedTopK在$k=10$时比标准堆快1--2\%，收益虽小但在百万级查询场景下可累积。

\subsection{SQ-SIMD：标量量化}

\subsubsection{动机与量化过程}

SQ（Scalar Quantization）的核心思想是用更紧凑的数据类型替代float32以降低内存占用和提升SIMD并行度。
对向量的每个维度独立进行min-max线性量化：

\begin{equation}
\text{code}[i][j] = \text{clamp}\left(\left\lfloor\frac{x_i[j] - \min_j}{\text{scale}_j} + 0.5\right\rfloor, 0, 255\right)
\end{equation}
其中$\min_j = \min_i x_i[j]$, $\max_j = \max_i x_i[j]$, $\text{scale}_j = (\max_j - \min_j) / 255$。

\subsubsection{数值分析}

DEEP100K: $N=100000$, $d=96$。
\begin{itemize}[nosep]
    \item float32 base: $100000 \times 96 \times 4 = 38.4$~MB
    \item uint8 base: $100000 \times 96 \times 1 = 9.6$~MB，压缩比 4$\times$
    \item 128-bit SIMD: 一次处理16个uint8（vs 4个float32），4$\times$并行度提升
    \item 查询LUT: $d\times 256\times 4 = 96$~KB（每query重建，无法完全放入48KB L1D）
\end{itemize}
rerank候选数$p$与精排访存量：$p=100\to 38.4$~KB, $p=1000\to 384$~KB, $p=5000\to 1.92$~MB。

\subsubsection{查询流程详解}

SQ查询分两阶段：(1) Coarse search——对每个query构建$d\times 256$的float LUT
（$\text{LUT}[j][c]=(\min_j+\text{scale}_j\cdot c)\cdot q[j]$），
遍历base code时通过查表累加得到近似内积，维护top-$p$候选列表。
(2) Rerank——对top-$p$候选，用原始float向量和NEON/AVX精确内积重算距离，
从中选出最终top-$k$。

\textbf{超参数$p$的权衡}：$p$过小则粗排精度不足，可能漏掉真正的最近邻（recall降低）；
$p$过大则精排访存量增大，延迟上升。$p$的选择需要根据目标recall在latency曲线上寻找
拐点——通常取recall刚好饱和的最小$p$值。

\subsubsection{U8SIMD路径的局限}

还实现了一条U8SIMD路径：query同样量化为uint8，coarse search用NEON整数dot
（\texttt{vmull\_u8 + vpadalq\_u16}）累加uint8内积。理论上能避免float LUT查表，
将coarse scan的每维操作从LUT访存+float加法变为整数乘加。
但实验表明recall极低（$p=2000$时仅0.275），因为uint8内积距离无法有效保持
float内积距离的顺序关系——量化误差在$d=96$维中累积后，距离排序被严重扰乱。
因此U8SIMD仅作为对照保留。

\subsection{PQ-SDC：对称距离计算}

\subsubsection{编码流程}

Product Quantization将向量空间分解为子空间并分别量化。编码分三步：将$d=96$切分为$M$个子空间（subdim$=d/M$）；
每子空间取2048条训练样本运行KMeans（$\text{Ks}=256$中心，10次迭代），得到$M$组codebook；
每向量各子空间以最近中心ID表示，subdim个float$\to$1个uint8（压缩比$4\times\text{subdim}$，$M=16$时整向量从384B压至16B）。

\subsubsection{SDC距离计算与误差分析}

SDC（Symmetric Distance Computation）对query也进行PQ编码，
并预计算每个子空间的中心间内积距离表：
\begin{equation}
D_m[c_q][c_b] = \langle C_m[c_q], C_m[c_b] \rangle, \quad c_q, c_b \in [0, 255]
\end{equation}
该表大小为$M\times 256\times 256\times 4$字节（$M=16$时4.0~MB），在索引构建时一次性计算。
查询时对$N$条base向量各做$M$次查表累加：
\begin{equation}
\hat{\delta}_{\text{SDC}}(q, x_i) = 1 - \sum_{m=0}^{M-1} D_m[\text{qcode}[m]][\text{base\_code}[i][m]]
\end{equation}

SDC误差来自双重量化：base端$\epsilon_b$和query端$\epsilon_q$。$M$较小时subdim大、聚类表达强但code少；
$M$较大时以更多code换取精度——$M$是SDC的核心超参数。

\subsection{PQ-ADC：非对称距离计算}

\subsubsection{算法原理}

ADC不编码query，每query在线构建局部LUT：
$\text{LUT}_m[c] = \langle q^{(m)}, C_m[c] \rangle, c \in [0, 255]$。
LUT构建需$M\times 256\times\text{subdim}$次乘加（$M=16$时仅24K次，$\approx 0.01$~ms）。
base扫描时查LUT累加：$\hat{\delta}_{\text{ADC}}(q, x_i) = 1 - \sum_m \text{LUT}_m[\text{base\_code}[i][m]]$。

\subsubsection{ADC vs SDC对比}

ADC的核心优势是非对称性：query保持原始float精度，仅base端量化，
因此recall系统性高于SDC。此外ADC不需要4.0~MB的SDC中心间距离表，
索引更紧凑。单query场景下两者的总计算量相当（ADC需LUT构建，SDC需query编码+2D查表）。

\subsection{SDC流水线并行}

SDC的query编码阶段（在$M$个子空间中各搜索最近中心）与scan查表阶段可以重叠执行。
在批量查询场景下，使用生产者-消费者模型实现流水线：

\[\text{Batch }t: \text{Stage 2 scan} \quad |\quad \text{Batch }t+1: \text{Stage 1 encode} \quad |\quad \text{Batch }t-1: \text{Stage 3 rerank}\]

实现了2-stage（encode与scan+rerank分离）和3-stage（三者完全分离）两种流水线，
使用有界阻塞队列（\texttt{BlockingQueue}）控制各阶段间的背压（back-pressure），
当队列满时生产者阻塞等待消费者。

流水线的加速潜力取决于各阶段的时间占比。在本项目中，
query编码阶段极短（$\approx 0.03$~ms, 占总延迟不到1\%），
scan阶段占$\approx 91\%$，因此即使完全重叠编码时间，理论加速上限仅为$1/(1-0.01)\approx 1.01\times$。
实测加速约6\%，与理论分析一致。SDC流水线在编码较重的场景（如复杂预处理）下更具价值。

\subsection{FastScan：查表累加阶段优化}

\subsubsection{设计动机}

普通PQ-ADC的scan阶段每query执行$N\times M = 100000\times 16 = 1.6\times 10^6$次
LUT查表和浮点累加。虽然每次查表的数据（float LUT, 16~KB for $M=16$）理论上在L1 cache内，
但随机访问模式（code值随机分布）导致L1访问延迟成为瓶颈。
FastScan通过三个技术优化scan阶段：

\begin{enumerate}[nosep]
    \item \textbf{LUT量化}：将float LUT全局量化为uint8，LUT大小从16~KB降至4~KB，
          显著降低L1 cache压力（4~KB仅为x86 L1D 48KB的8.3\%）
    \item \textbf{block scan}：将base code按block（$B=16\sim128$条向量）重新组织，
          使同一block内的候选共享LUT加载
    \item \textbf{累加方式优化}：用整数累加替代浮点累加，减少浮点加法延迟（3--5 cycles $\to$ 1 cycle）
\end{enumerate}

LUT量化公式：
$\text{qlut}[i] = \text{clamp}(\lfloor(\text{lut}[i] - \min)/\text{scale} + 0.5\rfloor, 0, 255)$。
量化引入的MAE误差通常在$10^{-3}\sim 10^{-4}$量级，通过后续float rerank阶段有效校正。

\subsubsection{v1 $\to$ v2 迭代：内存RMW的消除}

FastScan的实现经历了两版迭代，这是本文的核心工程贡献。

\textbf{v1（part-major布局）}：codes按\texttt{[block][part][lane]}组织——
同一part的所有lane的code连续存放。扫描时外层part、内层lane：
\begin{lstlisting}
for (part = 0; part < M; ++part)
    for (lane = 0; lane < B; ++lane)
        scores[lane] += qlut[part*256 + codes[lane]];  // memory RMW!
\end{lstlisting}
这种布局对SIMD gather友好（同一part的16条lane的code连续，可用\texttt{\_mm\_loadu\_si128}一次加载），
但累加目标\texttt{scores[lane]}在内存中，每次更新需三次L1操作：
(1) 读\texttt{scores[lane]}、(2) 读\texttt{qlut[...]}、(3) 写回\texttt{scores[lane]}。
对$M=16$, $B=64$，每block产生$16\times 64=1024$次内存RMW。
此外每block需\texttt{std::fill(scores, 0)}（$64\times 4=256$字节无效写）。

在ARM Kunpeng-920上，由于in-order流水线对cache miss敏感，
v1仍将scan从3.64降至2.75~ms（24.4\%加速）——
block-major的code布局改善了空间局部性。但在x86 i9-14900HX上，
OoO引擎可以很好地隐藏naive scan中LUT随机访问的L1延迟，
而v1引入的额外内存RMW反而造成38\%的性能倒退（1.87 vs 1.17~ms）。

\textbf{v2（lane-major布局 + 寄存器累加）}：codes改为\texttt{[block][lane][part]}，
每条lane的$M$个code连续。外层lane、内层part，累加在寄存器中：
\begin{lstlisting}
for (lane = 0; lane + 4 <= B; lane += 4) {  // 4-way unroll
    const uint8_t *c0 = codes + lane*M, *c1 = c0+M, *c2 = c1+M, *c3 = c2+M;
    int s0=0, s1=0, s2=0, s3=0;  // REGISTERS — zero memory RMW
    for (part = 0; part < M; ++part) {
        s0 += qlut[part*256 + c0[part]];  // L1 read + register add only
        s1 += qlut[part*256 + c1[part]];
        s2 += qlut[part*256 + c2[part]];
        s3 += qlut[part*256 + c3[part]];
    }
    coarse[lane+0] = {-(float)s0, id0};  // one-time store
    // ... store lane+1, +2, +3
}
\end{lstlisting}
v2的关键改进：
\begin{itemize}[nosep]
    \item 消除内存RMW：每条lane的score在32-bit整数寄存器中累加，仅最终写入内存一次
    \item 4路展开：4条lane的LUT查表相互独立，x86 OoO引擎可并行执行4条流水线
    \item 消除\texttt{std::fill}：score在寄存器中初始化为0，无需预清零内存
    \item uint8 LUT仅需256B/part（总计4KB for $M=16$），完全在L1内
\end{itemize}

\subsubsection{pshufb SIMD查表为何未采用}

理想FastScan应使用x86 \texttt{pshufb}（ARM等价的\texttt{vqtbl1q\_u8}）一次完成16个8-bit LUT查表。
但该指令仅使用低4位作为索引（每次查16-entry表），需将Ks降至16才能直接使用。

为此我们实现了v3版本：M=32个子空间、Ks=16，总code大小保持16 bytes/vector不变。
在x86上pshufb路径获得0.91~ms（相对v2的1.06~ms提速16\%），
在ARM上NEON vtbl路径获得2.20~ms（相对v2的2.44~ms提速10\%）。
然而\textbf{ARM上recall从0.99995降至0.996}，即使增大rerank候选数$p$也无法恢复至目标精度。
根因是$Ks=16$时每子空间仅16个聚类中心，在subdim=3的极低维空间中
无法有效表达向量分布，粗排阶段漏排导致精排无法挽救。

对256-entry LUT使用pshufb需拆分为16组nibble匹配（实测比标量慢15$\times$），
因此v3方案在追求recall$\geq$0.99995的目标下被放弃，最终采用v2的Ks=256标量lane-major方案。

\subsubsection{Block size超参数分析}

Block size $B$控制每block内的向量数量，影响L1 cache利用率：
\begin{itemize}[nosep]
    \item $B=16$：scores数组仅64字节，但block数量多（6250个），block切换频繁
    \item $B=64$：scores数组256字节，block数量1563，L1命中率最优
    \item $B=128$：scores数组512字节，开始超出部分L1容量，cache miss增加
\end{itemize}
ARM上$B=16$最优（2.16~ms），因Kunpeng-920的L1D为64KB，更小的block减少cache污染。
x86上$B=128$最优（1.06~ms），因i9-14900HX P-core的L1D为48KB且OoO更激进，
更大的block摊销了block切换开销。

(3) x86上loop unroll（2.15$\times$）反而不如单累加器AVX2（2.39$\times$），

(4) Prefetch在x86上有12\%加速（1.49$\to$1.31~ms, distance=64），
说明即使是i9-14900HX的激进硬件prefetcher，对大跨度（96$\times$4=384字节stride）的

因Kunpeng-920的prefetch引擎已经很好地覆盖了顺序stride访问。

(5) FixedTopK在双平台均获$\approx$1\%微加速，收益来自消除堆的$O(\log k)$调整开销
（对于$k=10$约为3--4次比较/交换）。

\subsection{x86 vs ARM 性能差异根因}

x86绝对性能领先ARM约3--5$\times$（Flat-Scalar: 3.56 vs 19.08~ms, 5.36$\times$）。
根因分解：
\begin{enumerate}[nosep]
    \item \textbf{时钟频率}：i9-14900HX P-core boost 5.8GHz vs Kunpeng-920 $\approx$2.6GHz（2.2$\times$）
    \item \textbf{SIMD位宽}：AVX2 256-bit vs NEON 128-bit，单指令数据吞吐2$\times$
    \item \textbf{内存带宽}：DDR5-5600双通道$\approx$89.6~GB/s vs DDR4-2666四通道$\approx$85.3~GB/s
          （理论值接近，但x86的实际有效带宽因更好的prefetcher和内存控制器而更高）
    \item \textbf{微架构}：x86宽OoO（512-entry ROB, 8-wide decode）vs ARM窄OoO/in-order，
          对L1 cache miss的容忍度差异显著——这是FastScan v1在ARM有效而在x86倒退的根因
\end{enumerate}

\section{总结}
% ===================================================================

本文围绕DEEP100K上ANN查询的距离计算优化，在ARM Kunpeng-920和x86 i9-14900HX双平台上
完整实现了Flat-SIMD（含scalar/auto-vec/manual/unroll/prefetch/top-k六种消融）、
SQ-SIMD（float LUT + U8SIMD两条路径）、PQ-SDC、PQ-ADC（含nth\_element Top-p选择）、
SDC流水线并行（2/3-stage + batch扫描）和FastScan（v1$\to$v2迭代）六类算法，
覆盖全部强制实验矩阵。手写SIMD vs 自动向量化对比、loop unrolling效果、
top-k维护优化和prefetch/alignment分析均已完成。

\textbf{优化路径}：
Flat-Scalar（ARM 19.08ms）$\xrightarrow{\text{SIMD}}$ Flat-NEON+Unroll（7.20ms, 2.65$\times$）
$\xrightarrow{\text{PQ}}$ PQ-ADC（3.69ms, 5.17$\times$）$\xrightarrow{\text{FastScan v2}}$ 最终（2.44ms, 7.82$\times$）。

\textbf{核心贡献}：FastScan v1$\to$v2的代码布局优化——识别出part-major布局的内存RMW瓶颈，
将其改为lane-major布局配合寄存器累加和4路展开。v2在ARM上相对v1额外加速22\%，
同时修复了v1在x86上的38\%性能倒退，使FastScan成为双平台一致的全局最优方案。
perf数据（IPC 0.96$\to$1.71$\to$1.94, L1D miss 2.70\%$\to$0.88\%$\to$0.65\%）
量化验证了从内存密集型到计算密集型的负载特征转变。

\textbf{教训}：(1) cache友好的内存布局必须与访问模式（标量累加 vs SIMD gather）匹配；
(2) 跨平台验证至关重要——OoO能力差异可以完全改变优化策略的效果；
(3) 对于内存密集型负载，消除不必要的内存RMW往往比提升计算并行度更有效。

项目代码见Git仓库，ARM平台通过\texttt{bash test.sh 1 1}可完整复现。
